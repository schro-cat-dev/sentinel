# タスク自動生成・承認フロー設計

> **ステータス**: 基本設計。Task Layer（ログ→タスク自動生成）と承認ワークフローは実装済み。
> インシデント対処フローとエスカレーションの詳細は以下の通り。

---

## 基本フロー

```
ログ受信
  → イベント検知（ルールベース）
  → タスク自動生成（ルール照合 + 重大度分類）
  → 実行レベルに応じた分岐:
      AUTO       → 即時実行
      SEMI_AUTO  → 承認リクエスト → 承認後に実行
      MANUAL     → 承認リクエスト → 人間が対応
      MONITOR    → 監視のみ（実行なし）
```

---

## 承認フロー

### 多段階承認チェーン

重大度に応じて承認チェーンの長さが変わる:

| 重大度 (LogLevel) | 承認ステップ | 承認者 |
|---|---|---|
| 1-3 (Trace/Debug/Info) | 1ステップ | チームリード |
| 4-5 (Warn/Error) | 2ステップ | チームリード → マネージャー |
| 6 (Critical) | 3ステップ | チームリード → マネージャー → CISO |

### 承認プロセスの要件

- **権限分離**: 承認者と実行者は異なる人物でなければならない
- **理由記録**: 各承認ステップで承認/却下理由を必須入力
- **コンテンツハッシュ**: 承認時にタスク内容のSHA-256ハッシュを記録し、実行時に照合（改ざん検知）
- **変更履歴**: 承認前にタスク内容が修正された場合、変更履歴がappend-onlyで記録される
- **不可逆性**: 一度承認/却下されたリクエストは変更不可

## 通知・エスカレーション

### 重大度に応じた通知先

| 重大度 | 通知先 | 方式 |
|--------|--------|------|
| LOW/INFO | 担当者 | Webhook |
| MEDIUM | 担当者 + チームリード | Webhook + Slack |
| HIGH | チームリード + マネージャー | Slack + メール |
| CRITICAL | 全関係者 + CISO | Slack + メール + PagerDuty |

### 超重大インシデント時の対処フロー

```
異常検知（エラー率急増、セキュリティ侵害兆候）
  ↓
1. キルスイッチ発動（全AIエージェント即時サスペンド）
2. 自動ログ隔離 + スナップショット取得（証跡確保）
3. 多層通知: 担当者 → 管理者 → 登録者（重要度に応じSlack/メール/電話）
  ↓
分類・エスカレーション:
  中:   Bさん確認後、再実行 or ロールバック。1回チェック。
  高:   登録者必須承認 + 2回チェック。AI提案を人間判断でフィルタ。
  超重大: 全自動化停止。インシデントレスポンスチーム召集。根本調査優先。
  ↓
復旧:
  人間承認でフィルタ通過タスクのみ段階再開。
  次回タスク生成時に「確認必須」フラグを自動付与。
```

## タスク実行レベルの事前指定

| カテゴリ | 実行レベル | 例 |
|---------|-----------|---|
| マイナーパッチ（事前登録済み） | AUTO | ログローテーション、キャッシュクリア |
| 確認後自動実行 | SEMI_AUTO | サービス再起動、設定変更 |
| 人間対応必須 | MANUAL | データ削除、権限変更、インフラ変更 |
| 監視のみ | MONITOR | 傾向分析、ベースライン計測 |
