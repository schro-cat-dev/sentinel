# 作業ログ — 2026-03-27 セキュリティモジュール群 拡張・実装

## 概要

Sentinel v1 (453テスト) → v2 (798テスト) への大規模拡張。
セキュリティモジュール群のサブモジュール導入・アンサンブル化・脅威レスポンス・通知アダプタ・SDK Transport抽象を一貫実装。

---

## 作業フェーズ

### Phase 1: セキュリティモジュール群のサブモジュール導入

**目的**: 既存のfirst-match検知を全ルール評価のアンサンブルに拡張し、検知精度と柔軟性を向上。

**実装内容:**

| 新規ファイル | 内容 |
|---|---|
| `detection/ensemble.go` | アンサンブル検知器 — ScoredDetectionRule I/F、ScoreAggregator(Max/Avg/WeightedSum)、ThresholdPolicy、PriorityResolver、既存ルールのWrapRuleアダプタ |
| `detection/dynamic_rule.go` | 設定ベース動的ルール — LogType/Level/MessagePattern/Tags/Origin/IsCriticalの条件組み合わせ、PayloadBuilder |
| `detection/anomaly.go` | 統計的異常検知 — FrequencyTracker(スライディングウィンドウ)、ベースライン比較、乖離率閾値 |
| `detection/dedup.go` | 時間ウィンドウ重複抑制 — 同一EventName|Boundary|ServiceIDの連続発火抑制 |
| `security/masking_policy.go` | コンテキスト依存マスクポリシー — ログType/Origin/Level別にマスクルール動的選択 |
| `security/masking_verify.go` | マスク後PII残留検証 — 全フィールド検査、preserve-awareフォールバック再マスク |
| `middleware/authorizer.go` | RBAC認可 — ロール→権限マッピング、AllowedLogTypes/DeniedLogTypes/MaxLogLevel |
| `domain/event.go` | DetectionResultにScore/RuleID/Suppressed追加、EnsembleResult型、AnomalyPayload |

**テスト:** 各モジュール15-25テスト、合計120+テスト追加

---

### Phase 2: 検知→AIエージェント委任パイプライン

**目的**: 検知されたタスクをAIエージェントに自動委任する仕組みの構築。

**実装内容:**

| 新規ファイル | 内容 |
|---|---|
| `engine/agent_bridge.go` | AgentBridge — TaskExecutorにAI_ANALYZEハンドラ登録、severity/actionフィルタ、ソースログキャッシュ |

**Pipeline拡張 (`engine/pipeline.go`):**
- PipelineConfigに15の新フィールド追加（Enable系フラグ + 各モジュール設定）
- Pipeline structに6つの新モジュール参照追加
- Process()を10ステージに拡張: Authorization → Normalize → Mask(Policy) → Verify → HashChain → Persist → Detect(Ensemble) → Anomaly → ThreatResponse → TaskGenerate+Dispatch

---

### Phase 3: 脅威レスポンスオーケストレーション

**目的**: 検知→即座ブロック→AI分析→通知の統合制御。設定ベースの戦略パターン。

**実装内容:**

| 新規ファイル | 内容 |
|---|---|
| `response/strategy.go` | ResponseStrategy(5種)、ThreatTarget抽出、ResponseRuleConfigマッチング |
| `response/block_agent.go` | BlockAction I/F、IPBlockAction、AccountLockAction、BlockDispatcher |
| `response/block_provider.go` | EnhancedBlockDispatcher(承認待ちモード)、AWS/GCP/AzureBlockAction |
| `response/analysis_agent.go` | AnalysisAgent I/F、AIAnalysisAgent、MockAnalysisAgent |
| `response/orchestrator.go` | ThreatResponseOrchestrator — 戦略→分析→ブロック→永続化→通知 |

**5つの戦略:**
- BLOCK_AND_NOTIFY: 分析→ブロック→通知
- ANALYZE_AND_NOTIFY: 分析→通知
- NOTIFY_ONLY: 通知のみ（デフォルト）
- BLOCK_ONLY: サイレント防御
- MONITOR: ログ記録のみ

---

### Phase 4: 通知アダプタパターン

**目的**: Slack/Gmail/Discord/Webhook等の通知先をアダプタパターンで拡張可能に。

**実装内容:**

| 新規ファイル | 内容 |
|---|---|
| `notify/notifier.go` | Notifier I/F、MultiNotifier(プレフィックスベースルーティング) |
| `notify/adapters.go` | WebhookNotifier(HMAC署名)、SlackNotifier(色分けattachment)、GmailNotifier(SMTP)、DiscordNotifier(embed)、LogNotifier |
| `notify/mock.go` | MockNotifier(テスト用) |

---

### Phase 5: Config拡張 + main.goワイヤリング

**目的**: 全新モジュールを設定ファイル/環境変数で制御し、本番で動作させる。

**config.go拡張:**
- AgentConfig, EnsembleConfig, AuthorizationConfig, AnomalyDetectionConfig, ResponseConfig
- 環境変数: SENTINEL_AGENT_ENABLED, SENTINEL_ENSEMBLE_ENABLED, SENTINEL_AUTHZ_ENABLED, SENTINEL_ANOMALY_ENABLED, SENTINEL_RESPONSE_ENABLED, SENTINEL_RESPONSE_DEFAULT_STRATEGY

**main.go改修:**
- Config→PipelineConfig変換関数7つ: convertDynamicRules, convertAggregator, convertAuthzConfig, convertAllowedActions, convertResponseRules等
- Post-init: AgentBridge接続、ThreatOrchestrator接続（persistFn→store.InsertThreatResponse、notifyFn接続）
- StartServerWithSentinel() + Pipeline()アクセサでpost-init設定可能

---

### Phase 6: Proto再生成 + Store拡張 + 実接続

**目的**: ThreatResponseをgRPCレスポンスで返し、DBに永続化する。

**変更:**
- `proto/sentinel.proto`: ThreatResponseSummaryメッセージ追加、IngestResponseにthreat_responsesフィールド追加
- `pb.go`: protoc再生成
- `server.go`: ThreatResponseSummaryマッピング実装
- `store/store.go`: InsertThreatResponse/GetThreatResponsesByTraceID追加
- `store/sqlite.go`: threat_responsesテーブル + INSERT/SELECT実装

---

### Phase 7: SDK Transport抽象

**目的**: SDKがGoサーバにリモート送信できるようにする（zero-dep維持）。

**実装:**
- `transport/transport.ts`: RemoteTransport I/F、TransportMode(local/remote/dual)、TransportConfig
- `index.ts`: SentinelOptions追加、ingest()でmode分岐（remote/dual/local）
- 7テスト追加: remote成功/失敗/フォールバック、dual成功/失敗、criticalタスク生成

---

### Phase 8: 監査 + テスト品質修正

- Ensemble空ルールテスト不足
